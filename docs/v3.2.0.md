## 배경

v3.1.0에서 도입된 Bedrock GPU 모니터링 에이전트를 운영하면서 발견된 문제점 3가지 개선:

1. **Loki 연동 부재**: Lambda가 Prometheus 메트릭만 수집하고 애플리케이션 로그를 참조하지 못해 Bedrock 분석 품질 저하
2. **Triton 메트릭 트리거 오탐**: DCGM에 Triton 메트릭이 없는 환경에서 해당 부재 자체가 알림을 유발하는 문제
3. **Cooldown 미적용**: 동일 장애가 지속될 경우 매 트리거마다 중복 Slack 알림 발생

## ✏️ Changed

### lambda_src/gpu_monitor/lambda_function.py

#### 1. Loki 로그 수집 연동

Prometheus 메트릭 수집과 함께 Loki에서 최근 애플리케이션 로그를 조회하여  
Bedrock에 컨텍스트로 함께 전달. 메트릭 이상과 로그 패턴을 결합한 분석 가능.

```
수집 데이터:
  Before: GPU 메트릭 (Prometheus)
  After:  GPU 메트릭 (Prometheus) + 애플리케이션 로그 (Loki)
```

#### 2. Triton 메트릭 트리거 삭제

laurel GPU 서버 환경에서 Triton Inference Server 메트릭이 수집되지 않는 구조임을 확인.  
해당 메트릭 부재를 이상으로 판단하던 트리거 조건 제거.

#### 3. Cooldown 로직 추가

동일 알림이 반복 발송되는 문제를 방지하기 위해 알림 발송 후 일정 시간 동안  
중복 알림을 억제하는 Cooldown 메커니즘 추가.

> Cooldown 구현 상세는 `docs/bedrock_agent/overview.md` 참고.

## 운영 참고

- Loki 연동으로 Lambda 실행 시간 증가 가능 → CloudWatch 로그에서 Duration 모니터링 권장
- Cooldown 상태는 Lambda 메모리 내 관리 (재배포 시 초기화)
- Triton 관련 메트릭이 향후 추가되면 트리거 조건 재추가 필요
