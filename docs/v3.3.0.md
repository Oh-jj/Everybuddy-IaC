# v3.3.0 — S3 Data Lake 도입 (GPU 에러 이력 저장)

**날짜:** 2026-06-15  
**커밋:** `5bad83e` · `(fix commits)`

---

## 배경

Bedrock GPU 모니터링 에이전트(v3.1.0~)가 이상 감지 시 Slack 알림만 발송하고 구조화된 이력을 남기지 않는 문제 개선.  
동일 에러가 반복돼도 매번 초기 분석만 수행되며, 운영 중 에러 패턴 집계가 불가능한 상태였음.

---

## ✨ Added

### modules/datalake/

S3 기반 Data Lake 모듈 신규 추가.

| 리소스 | 설명 |
| --- | --- |
| `aws_s3_bucket` | `everybuddy-datalake-{suffix}` 버킷 생성 |
| `aws_s3_bucket_public_access_block` | 퍼블릭 접근 전면 차단 |
| `aws_s3_bucket_policy` | Lambda Role ARN에 `s3:PutObject` 허용 (버킷 정책으로 IAM 수정 없이 처리) |
| `aws_s3_bucket_lifecycle_configuration` | 365일 후 Glacier 전환 |

**저장 경로 구조:**
```
s3://everybuddy-datalake-{suffix}/
  laurel-errors/
    year=YYYY/month=MM/day=DD/
      {unix_timestamp}_{SEVERITY}.json
```

**저장 레코드 스키마:**
```json
{
  "timestamp": "2026-06-15T13:00:00+00:00",
  "severity": "CRITICAL | WARNING | LOG_ONLY",
  "issues": ["GPU 0 온도 초과", "..."],
  "loki_errors": ["[ERROR] CUDA OOM", "..."],
  "metrics_snapshot": {
    "temps":  {"0": 85.0},
    "utils":  {"0": 98.0},
    "vrams":  {"0": 99.5},
    "powers": {"0": 320.0}
  },
  "bedrock_analysis": "Bedrock 분석 결과 텍스트"
}
```

## ✏️ Changed

### lambda_src/gpu_monitor/lambda_function.py

이상 감지 후 Bedrock 분석 완료 시점에 `save_to_datalake()` 호출 추가.

- `WARNING` / `CRITICAL` (메트릭 이상 + 쿨다운 통과 시)
- `LOG_ONLY` (메트릭 정상, 에러 로그만 존재 시)

S3 저장 실패는 예외를 흡수하여 Slack 알림 발송에 영향 없음.

### main.tf / variables.tf

- `datalake` 모듈 추가
- `bedrock_agent` 모듈에 `datalake_bucket_name` 전달
- `datalake_bucket_suffix` 변수 추가 (default 없음 — 환경변수로 주입)

### modules/bedrock_agent/

- `DATALAKE_BUCKET` 환경변수 Lambda에 주입

## 운영 참고

- **terraform apply 전** 환경변수 추가 필요:
  ```bash
  export TF_VAR_datalake_bucket_suffix="<계정ID 앞 6자리 등 고유값>"
  ```
- S3 저장은 Lambda 메모리와 무관하게 독립 실패 → 저장 오류 시 CloudWatch 로그에서 `[ERROR] Data Lake 저장 실패` 확인
- 향후 Athena + Partition Projection으로 SQL 집계 가능한 구조로 설계됨 (데이터 축적 후 도입 예정)
